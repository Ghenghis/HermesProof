The problem

Three agents. One repository. Zero coordination.

You've wired Claude, Codex and Windsurf into the same workspace. They're powerful — but they don't know about each other. The first time two of them touch the same file at the same time, things break silently.

Race conditions

Two agents save the same file within milliseconds. The second write silently overwrites the first. No warning, no log, no recovery.

No audit trail

Who edited what, when, and on whose authority? Standard MCP servers don't say. You're left reading git log and guessing.

Coordination chaos

"Hand this off to Codex" is a vibe, not a protocol. Without explicit ownership, agents over-edit each other or freeze waiting for permission that never comes.

The solution

One MCP server. Three guarantees. Every push re-proven.

Atomic per-file locks

Acquired via mkdir(EEXIST) on a content-hashed directory. Two agents racing for the same file: one wins, the other gets a structured "blocked" with the current owner. Always.

Append-only evidence

Every state change — lock acquired, handoff requested, gate run — appended to two NDJSON ledgers with stable IDs and timestamps. You can replay the entire session.

Truth-gate proof bundle

Nine independent verifications run on every push. If any required gate fails, CI fails. If all pass, a fresh PROOF/latest.json is committed back. The repo proves itself.

End-to-end

Six stages. Immutable evidence at every one.

An edit travels from intent to attestation through six logical stages. Each writes append-only evidence; nothing is silently mutated.

Trigger Bridge — v0.4

State changes become durable file events. The bridge is passive.

Every coordination state change — lock claimed, handoff approved, evidence appended, gate result, PR opened — emits a JSON envelope into events/outbox/. A local watcher polls, generates a deterministic review packet, and atomically renames the file into events/handled/ when the consumer acknowledges. Schema-mismatched events go to events/failed/. HermesProof never calls an LLM — the file is the trigger surface; an external runner, webhook, or human reads the outbox.

4 new MCP tools

hermes_emit_event · hermes_list_events · hermes_mark_event_handled · hermes_create_blocked_handoff — total tool count is now 20.

3 watcher modes

A. Console — print event lines. B. Review packet — deterministic Markdown to review_packets/. C. Webhook — POST envelope to HERMESPROOF_WEBHOOK_URL. No paid services required.

5 architect invariants

Loop guard on evidence.appended · atomic fs.rename for state moves · retention only on handled/ · required event_schema_version: 1 · evidence_ids resolved at emit time.

GitHub Actions mechanical PR review

A new workflow .github/workflows/hermesproof-review-check.yml runs on every PR open without an LLM: asserts evidence-chain line, task id, gate run, changed-file count, valid handoff (if any handoff is added), PROOF/latest.json presence, and README tool count matches the server's registered tool count. Catches lock-discipline drift mechanically.

Lock lifecycle

A four-state machine, deliberately minimal.

Each tracked file is in exactly one of four states. Transitions are atomic, observable, and — except for stale recovery — require explicit consent.

State transitions

  • UNLOCKED → HELD(A)  lock_files wins via mkdir EEXIST
  • HELD(A) → HELD(A)  heartbeat extends TTL while work continues
  • HELD(A) → HANDOFF_PENDING  B calls request_handoff
  • HANDOFF_PENDING → HELD(B)  A calls approve_handoff(approve)
  • HANDOFF_PENDING → HELD(A)  A calls approve_handoff(deny)
  • HELD(*) → UNLOCKED  release_files by owner, or recover_stale_locks after TTL

Why so few states?

Every additional state is a place for a bug to hide. HermesProof has exactly the states required to prove the safety properties:

  • Mutual exclusion — at most one holder at any time.
  • Explicit transfer — ownership moves only via approved handoff.
  • No silent override — recovery requires a TTL expiry plus an explicit call, both logged.
  • Auditability — every transition is a line in events.ndjson.

FAQ

Questions you'd actually ask.

Why not just use file system locks (flock, LockFileEx)?

Filesystem locks are tied to a process and are advisory on every major OS. An agent that crashes leaves a stuck lock; an agent that doesn't check the lock just walks past it. HermesProof keeps the lock state in a separate, observable directory with TTL metadata, so any agent can list locks at any moment and know exactly who owns what.

You also can't audit a kernel lock. Every HermesProof transition is a line in events.ndjson — replayable, diffable, attestable.

Does this slow agents down?

No. The hot path is one mkdir + one writeFileSync on the agent's own machine. Sub-millisecond. The slow part of multi-agent work was always coordination overhead — the time wasted re-doing each other's edits, fixing merge conflicts, asking "are you done?". HermesProof removes that.

What happens if an agent crashes mid-edit?

The lock metadata stays on disk. Other agents see it via list_locks, including its expires_utc. Once the TTL passes (default 90 minutes), any agent can call recover_stale_locks. The recovery itself is logged. There is no silent override path.

Can it run arbitrary shell commands?

No. hermes_run_gate only accepts gate IDs from a hardcoded allowlist (git-status, npm-test, npm-audit, etc.). The actual command + args are constants in source. spawn is called with shell: false. There is no path for user-supplied commands or arguments to reach spawn.

Is it project-specific?

Despite the name, no. The deployed MCP server name is hermes3d-locks (preserved for already-wired clients), but the package itself works against any workspace. Just point MCP_LOCK_WORKSPACE at the repo you want to govern. See SETUP_GENERIC_PROJECT.md.

How is it different from Hermes Agent (Nous Research)?

Hermes Agent reasons and acts. HermesProof governs and attests. Where Nous's hermes-agent uses tool-calling to drive an autonomous workflow, HermesProof is the layer that lets multiple Hermes-class agents cooperate on a real codebase without overwriting each other's work. Same lineage, different rung of the stack.

Why is the GitHub server name hermes3d-locks instead of hermesproof?

The MCP server name is what every connected client (Claude Desktop, Claude Code, Codex, Windsurf) recorded in its config the first time it was wired. Renaming would break all four installs. The package binary is hermesproof; the over-the-wire server name stays hermes3d-locks for backwards compatibility. New servers may want to start fresh.
