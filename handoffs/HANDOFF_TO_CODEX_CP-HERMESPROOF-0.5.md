# HANDOFF_TO_CODEX_CP-HERMESPROOF-0.5

> **Status:** READY for Codex to claim. Builds on v0.4 (trigger bridge) which is now live on `main`. Implements the file-based task queue + 4 new MCP tools so future cycles can `hermes_pick_task` from a pending pool instead of every brief being a paste-and-relay.

## 1. Mission

Add a durable, file-based **task queue** at `.hermes3d_orchestrator/tasks/{pending,claimed,blocked,done}/` so any agent can enqueue work for any other agent and pick the highest-priority owner-matching task on session start. Tasks emit v0.4 events on every state transition so watchers/webhooks fire automatically.

This unblocks the multi-agent loop's "pickup discipline" problem. Today, Claude writes a brief in `handoffs/`, the user pastes it into Codex's chat. With v0.5, Claude `hermes_enqueue_task`s the brief, and Codex's standing prompt does `hermes_pick_task` at session start.

> **Important truth:** v0.5 is still passive. HermesProof does not start chat sessions. The "auto-pickup" is a discipline pattern (Codex's standing prompt checks the queue first) — exactly the same way v0.4's trigger bridge is passive. Headless background runners that auto-start Codex sessions remain a future v0.7+ tier.

## 5. Required behavior

### 5.1 Queue lifecycle

```text
ENQUEUE  →  pending/<task_id>.json
PICK     →  pending/<task_id>.json  ──fs.rename──>  claimed/<task_id>.json
DONE     →  claimed/<task_id>.json  ──fs.rename──>  done/<task_id>.json
BLOCK    →  claimed/<task_id>.json  ──fs.rename──>  blocked/<task_id>.json
RECOVER  →  claimed/<task_id>.json (stale)  ──fs.rename──>  pending/<task_id>.json
```

All transitions use **`fs.rename` on the same filesystem** (atomic). Concurrent pickers get `task_already_claimed` from the loser, never double-pickup.

### 5.2 Task envelope

```json
{
  "task_schema_version": 1,
  "task_id": "H3D-CP5.1-C",
  "title": "...",
  "summary": "...",
  "handoff_path": "handoffs/HANDOFF_TO_CODEX_CP5.1-C.md",
  "branch_hint": "feat/...",
  "files_hint": [],
  "priority": 0,
  "target_owner_pattern": "^codex-impl-[0-9]+$",
  "enqueued_by": "claude-lead",
  "enqueued_utc": "2026-05-03T...",
  "claimed_by": null,
  "claimed_utc": null,
  "ttl_minutes": 120,
  "heartbeat_utc": null,
  "done_utc": null,
  "blocked_reason": null,
  "data": {}
}
```

`task_schema_version: 1` is REQUIRED. Unknown versions go to `blocked/` with `blocked_reason: "unknown_schema_version"` (mirrors v0.4 event handling).

### 5.3 Three architect specifics (not optional)

1. **Priority field**: tasks have `priority: number` (default 0). `hermes_pick_task` returns highest priority first; ties broken by `enqueued_utc` ascending (FIFO within priority class). Document this rule in `docs/QUEUE_PROTOCOL.md`.

2. **Owner-affinity matching**: tasks have `target_owner_pattern: string` (a regex). `hermes_pick_task` rejects with `task_owner_mismatch` if the calling owner doesn't match the regex. Default pattern `.*` (any owner). The regex MUST be pre-validated at `enqueue_task` time — invalid regex rejects with `invalid_owner_pattern`.

3. **Stale-task recovery**: tasks claimed past TTL are recoverable via `hermes_recover_stale_tasks`. Mirrors the existing `hermes_recover_stale_locks`. Tool emits `task.recovered` event (extend v0.4's EVENT_TYPES set). Default TTL: 120 min. Heartbeat-extending the claim is via `hermes_heartbeat` (existing tool, extended to also touch claimed task records).

### 5.4 Four new MCP tools

| Tool | Inputs | Annotations | Behavior |
|---|---|---|---|
| `hermes_enqueue_task` | `{task_id, title, summary, handoff_path?, branch_hint?, files_hint?, priority=0, target_owner_pattern=".*", ttl_minutes=120, data={}}` | non-readonly, non-destructive, idempotent (re-enqueue same task_id is no-op) | Validates regex, writes `pending/<task_id>.json` atomically, emits `task.enqueued` event |
| `hermes_list_pending_tasks` | `{owner_filter?, limit=50}` | readonly, idempotent | Returns sorted pending tasks (priority desc, then enqueued_utc asc); if `owner_filter` given, returns only tasks whose `target_owner_pattern` matches |
| `hermes_pick_task` | `{owner, prefer_task_id?}` | non-readonly, non-idempotent (claims one task) | Atomically picks highest-priority owner-matching task; rejects with `no_pending_tasks_for_owner` if none. Sets `claimed_by`, `claimed_utc`. Emits `task.claimed` event (already in v0.4 EVENT_TYPES). |
| `hermes_recover_stale_tasks` | `{owner, files=[]}` (parallels lock recovery API) | destructive (forces task release), idempotent | Moves expired claimed tasks back to pending; emits `task.recovered` event (NEW event type; extend EVENT_TYPES) |

`task.recovered` is added to `EVENT_TYPES` set in `event-manager.mjs`. Backward-compatible (additive enum).

### 5.5 Helper script: `scripts/next-task.sh`

POSIX shell wrapper for non-MCP clients. Calls into the MCP server via stdio JSON-RPC and outputs the picked task as JSON to stdout. Used by AGENTS.md-style standing prompts that don't have native MCP support yet.

```bash
scripts/next-task.sh --owner codex-impl-01 --prefer-task-id H3D-CP5.1-C
```

Uses `node`, no python/curl deps. Returns exit 0 + JSON on success, exit 1 + stderr message on failure.

### 5.6 Queue doctor: `scripts/queue-doctor.mjs`

Mirrors `scripts/trigger-doctor.mjs`'s shape. Verifies:
- `pending/`, `claimed/`, `blocked/`, `done/` exist and are writable
- A round-trip enqueue → pick → done works on a sandbox
- Owner-pattern rejection works (mismatch returns `task_owner_mismatch`)
- Priority ordering is correct (3 tasks priority 5/3/1 → picked in order)
- Stale recovery works (claim, fast-forward TTL, recover, re-pick)

Exit 0 = pass. Used by truth-gate `queue.doctor_passes`.

## 6. Required tests (extend `coordination-smoke-test.mjs`)

```text
- task envelope validates task_schema_version=1
- enqueue creates pending/<id>.json with priority + owner pattern fields
- list_pending returns sorted by priority desc + enqueued_utc asc
- pick atomically claims (concurrent pickers: only one wins, other gets task_already_claimed)
- pick respects target_owner_pattern (mismatch -> task_owner_mismatch)
- invalid owner_pattern at enqueue time rejected with invalid_owner_pattern
- enqueue is idempotent (re-enqueue same task_id with same fields = no-op success)
- task.enqueued event lands in events/outbox/ on enqueue
- task.claimed event fires on pick (already in v0.4 EVENT_TYPES)
- task.recovered event fires on recover (NEW event type)
- recover_stale_tasks ignores not-yet-expired claims
- task path cannot escape workspace (security)
```
