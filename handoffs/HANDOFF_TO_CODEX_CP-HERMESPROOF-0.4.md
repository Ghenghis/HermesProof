## 5. Required behavior

### 5.1 Events emitted

HermesProof MUST create durable events on these state transitions:

```text
task.claimed
task.released
task.blocked
handoff.created
handoff.approved
handoff.denied
lock.acquired
lock.released
lock.recovered
evidence.appended
gate.failed
gate.passed
pr.opened          (only when payload.pr_url provided by caller)
```

> **🔧 Improvement #1 (architect): infinite-loop guard.** The `evidence.appended` event must NOT be emitted for evidence rows that were themselves written by the event-manager's bookkeeping. Internal evidence writes (the row that documents an event was created) carry `data.system: "event-manager"` and the emitter MUST skip self-re-emission. Test this with a dedicated unit test — see §6.

### 5.2 Storage

```text
.hermes3d_orchestrator/events/outbox/<event_id>.json     pending consumption
.hermes3d_orchestrator/events/handled/<event_id>.json    after mark_handled
.hermes3d_orchestrator/events/failed/<event_id>.json     processing failed
```

> **🔧 Improvement #2 (architect): atomic moves.** All state transitions between outbox/handled/failed MUST use `fs.rename` (atomic on the same filesystem). Concurrent watchers that lose the race surface as `event_already_handled` errors, NOT as double-processing. Document this in `docs/EVENT_SCHEMA.md` §"Concurrency".

> **🔧 Improvement #3 (architect): retention policy.** Add `scripts/prune-events.mjs --before <iso> [--dry-run]` that deletes `events/handled/*.json` older than the cutoff. Default cutoff suggestion in MAINTENANCE.md: 30 days. **`events/failed/` is NEVER auto-pruned** — operators inspect manually. Add a CI gate (or scheduled cron in a follow-up PR) only after operators agree.

### 5.3 Event envelope

```json
{
  "event_schema_version": 1,
  "event_id": "evt_<utc_iso_compact>_<6char_sha>",
  "event_type": "task.released",
  "created_utc": "2026-05-03T00:00:00.000Z",
  "workspace_root": "G:\\Github\\Hermes3D",
  "task_id": "H3D-CP5.1-B",
  "owner": "codex-impl-01",
  "branch": "feat/...",
  "files": [],
  "summary": "...",
  "evidence_ids": [],
  "next_actor": "claude",
  "recommended_action": "review_pr",
  "payload": {}
}
```

> **🔧 Improvement #4 (architect): schema versioning.** `event_schema_version: 1` is required. Future evolutions bump the version. Consumers MUST honor the version field; unrecognized versions go to `failed/` with `error: "unknown_schema_version"`. Mirrors the doctor JSON envelope versioning pattern from CP5.1-C (see Hermes3D ADR-013 §3).

> **🔧 Improvement #5 (architect): `evidence_ids` semantics.** This field MUST contain the evidence-row IDs appended during the lifetime of the `task_id` referenced in the event, ordered by `prev_hash` chain. For events not bound to a task (`gate.failed` invoked outside a task context), the field is `[]`. Codex MUST resolve this list by walking the chain at event-emit time, not at event-handle time (a later watcher should not need to re-walk).

`next_actor` enum: `claude | codex | human | unassigned`.
`recommended_action` enum: `review_pr | fix_scope | merge | review_handoff | acknowledge | none`.

### 5.4 New MCP tools

| Tool | Input | Behavior |
|---|---|---|
| `hermes_list_events` | `{status: outbox|handled|failed|all, limit: int}` | Returns events in chronological order |
| `hermes_mark_event_handled` | `{event_id, handled_by, note}` | `fs.rename` outbox/→handled/, append non-emitting evidence |
| `hermes_emit_event` | full envelope (callers fill `event_type`, `next_actor`, `recommended_action`, `payload`; manager fills `event_id`, `created_utc`, `workspace_root`, etc.) | Manual event insertion |
| `hermes_create_blocked_handoff` | `{task_id, owner, reason, blocked_files, suggested_correct_paths, handoff_path, release_locks: bool}` | Writes handoff.md, emits `task.blocked`, appends evidence, optionally releases locks |

### 5.5 New scripts

**`scripts/watch-events.mjs`** — three modes:

- **Mode A (default)** — console-only: prints `[HermesProof] task.released H3D-CP5.1-B by codex-impl-01 → review_pr` and the path of any review packet generated.
- **Mode B (`--write-review-packets`)** — writes review packet via `generate-review-packet.mjs`, leaves event in outbox.
- **Mode C (`HERMESPROOF_WEBHOOK_URL` env set)** — POSTs the event JSON to the URL, default `Content-Type: application/json`, no auth header (operator adds via reverse-proxy if needed). On 2xx, optional `--mark-handled` flag moves event to handled.

Flags: `--workspace <path>`, `--poll <seconds>` (default 30), `--once`, `--write-review-packets`, `--review-packet-dir <path>`, `--mark-handled` (default false).

**`scripts/generate-review-packet.mjs`** — emits `.hermes3d_orchestrator/review_packets/REVIEW_<task_id>_<event_id>.md` with: event details, task id, owner, branch, files changed (from PR if available via `gh pr view`), evidence ids, PR number if in payload, gate status, and an exact ready-to-paste Claude review prompt.

**`scripts/trigger-doctor.mjs`** — verifies the trigger system end-to-end against a sandbox workspace. Exit 0 = pass.

**`scripts/prune-events.mjs`** — retention cleanup (Improvement #3).

### 5.6 GitHub Actions workflow — `.github/workflows/hermesproof-review-check.yml`

Triggers: `pull_request: [opened, synchronize, reopened, ready_for_review]`.

**Runs LLM-free mechanical checks. Must pass for the PR to be considered "review-ready":**


Annotates the PR with a single summary comment. Re-runs on every push.

## 11. Improvements over upstream contract (architect notes)

The downloaded contract `CP-HERMESPROOF-0.4_TRIGGER_BRIDGE_CODEX_CONTRACT.md` (good baseline; thanks for drafting it) is solid but has 5 gaps that this brief fixes:

1. **Infinite-loop guard** (§5.1) — without it, `evidence.appended` events recursively emit themselves (the event-manager appends evidence saying "event emitted" → fires `evidence.appended` → appends another row → …).
2. **Atomic move semantics** (§5.2) — `fs.rename` on same filesystem is the only safe primitive; doc + enforce.
3. **Retention** (§5.2) — `events/handled/` would grow unbounded; ship `prune-events.mjs` from day one.
4. **Event schema versioning** (§5.3) — mirrors the doctor JSON envelope pattern from CP5.1-C, lets v0.5+ evolve cleanly.
5. **evidence_ids semantics** (§5.3) — without spec, Codex would guess; resolve at emit time, ordered by chain.

All 5 are baked into §5 and tested in §6. The `Final report format` from the upstream contract is still authoritative; reuse it verbatim in the PR body.
