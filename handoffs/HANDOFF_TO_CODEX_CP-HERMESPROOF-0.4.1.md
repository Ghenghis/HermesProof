## 1. Mission

Address the three real (but non-blocking) issues Gemini flagged on the v0.4 trigger bridge:

1. **`evidenceIdsForTask` reads the entire NDJSON ledger into memory on every event emit** — O(N) memory + I/O per event, scales badly past ~10k entries. Fix: stream-based readline.
2. **`listEvents` reads + parses every JSON file in the event directories before applying `limit`** — wasteful especially on `handled/` which accumulates. Fix: sort filenames first (timestamps-prefixed = sortable), only read `limit` files.
3. **Webhook `fetch` in `watch-events.mjs` has no timeout** — an unresponsive endpoint hangs the watcher process indefinitely. Fix: `AbortController` with 10-second timeout.

These are quality-of-implementation issues. v0.4 functionally works (PR #5 shipped clean). v0.4.1 is the perf-and-robustness pass before v0.5 builds on top.

## 5. Required behavior — three fixes

### Fix #1 — `evidenceIdsForTask` stream-based read

Current implementation in `src/core/event-manager.mjs` (~lines 184-188 per Gemini's review):

```js
async evidenceIdsForTask(taskId) {
  const entries = await readEvidenceEntries(this.paths.evidenceFile);
  return entries
    .filter((entry) => entry && entry.task_id === taskId && typeof entry.entry_hash === "string")
    .map((entry) => entry.id)
    .filter(Boolean);
}
```

`readEvidenceEntries` reads the whole file into memory and parses every line. Replace with a `node:readline` stream:

```js
import readline from "node:readline";
import { createReadStream } from "node:fs";

async evidenceIdsForTask(taskId) {
  let stream;
  try {
    stream = createReadStream(this.paths.evidenceFile, { encoding: "utf8" });
  } catch (err) {
    if (err.code === "ENOENT") return [];
    throw err;
  }
  const rl = readline.createInterface({ input: stream, crlfDelay: Infinity });
  const ids = [];
  try {
    for await (const line of rl) {
      if (!line) continue;
      let entry;
      try { entry = JSON.parse(line); } catch { continue; }
      if (entry && entry.task_id === taskId && typeof entry.entry_hash === "string" && entry.id) {
        ids.push(entry.id);
      }
    }
  } finally {
    rl.close();
    stream.destroy();
  }
  return ids;
}
```

**Preserve order**: `readline` emits in file order = chain order on append-only ledger. Don't sort; the existing implementation also relied on insertion order.

`readEvidenceEntries` still has callers (`hermes_verify_evidence` chain walk in `lock-manager.mjs`), so keep the function exported but DO NOT call it from `evidenceIdsForTask` anymore.

### Fix #2 — `listEvents` filename-first lazy read

Current implementation (~lines 114-128):

```js
const events = [];
for (const value of statuses) {
  const dir = this.dirForStatus(value);
  const files = await fs.readdir(dir).catch(() => []);
  for (const file of files.filter((name) => name.endsWith(".json"))) {
    const event = await readJson(path.join(dir, file), null);
    if (event) events.push({ status: value, ...event });
  }
}
events.sort(...);
const bounded = ...;
return { events: events.slice(0, bounded) };
```

Rewrite to:

```js
const allFiles = [];
for (const value of statuses) {
  const dir = this.dirForStatus(value);
  const names = await fs.readdir(dir).catch(() => []);
  for (const name of names) {
    if (name.endsWith(".json")) {
      allFiles.push({ status: value, name, path: path.join(dir, name) });
    }
  }
}
// event_id format `evt_<utc_iso_compact>_<6hex>` is naturally sortable,
// so filename ordering = chronological ordering. No need to read+sort.
allFiles.sort((a, b) => a.name.localeCompare(b.name));
const bounded = Math.max(1, Math.min(500, Number(limit || 50)));
const events = [];
for (const item of allFiles.slice(0, bounded)) {
  const event = await readJson(item.path, null);
  if (event) events.push({ status: item.status, ...event });
}
return { ok: true, status, count: allFiles.length, events };
```

**Crucial**: `count` reflects the TOTAL number of matching files (not just the returned ones), so consumers know when more events exist beyond `limit`. The current code's `count` was equal to `events.length`, which silently underreports when limit < total. Fix that semantic too.

### Fix #3 — webhook timeout via `AbortController`

Current implementation in `scripts/watch-events.mjs` (~lines 48-56):

```js
const response = await fetch(webhookUrl, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(event)
});
posted = response.status >= 200 && response.status < 300;
console.log(`[HermesProof] webhook ${event.event_id} status=${response.status}`);
```

Wrap in `AbortController` with a 10-second default timeout (env-overridable):

```js
const TIMEOUT_MS = Number(process.env.HERMESPROOF_WEBHOOK_TIMEOUT_MS || 10000);

const controller = new AbortController();
const timer = setTimeout(() => controller.abort(), TIMEOUT_MS);
try {
  const response = await fetch(webhookUrl, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(event),
    signal: controller.signal
  });
  posted = response.status >= 200 && response.status < 300;
  console.log(`[HermesProof] webhook ${event.event_id} status=${response.status}`);
} catch (err) {
  posted = false;
  if (err.name === "AbortError") {
    console.error(`[HermesProof] webhook timeout ${event.event_id} after ${TIMEOUT_MS}ms`);
  } else {
    console.error(`[HermesProof] webhook failed ${event.event_id}: ${err.message}`);
  }
} finally {
  clearTimeout(timer);
}
```

The `AbortError` path keeps the watcher loop alive (no rethrow). The event stays in `outbox/` (`mark_handled` only fires on 2xx success), so a transient webhook outage doesn't lose events.
