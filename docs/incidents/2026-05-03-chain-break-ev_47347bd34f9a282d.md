# Incident: Evidence Chain Break at ev_47347bd34f9a282d

**Date:** 2026-05-03  
**Incident ID:** `chain-break-ev_47347bd34f9a282d`  
**Status:** accepted historical fork; no in-place hash rewrite  
**Related audit item:** P0-3 in `docs/audits/2026-05-03/AUDIT_2026-05-03.md`

## Summary

`hermes_verify_evidence` reports the first broken link in the local HermesProof evidence ledger at `ev_47347bd34f9a282d`. The entry is not malformed and its own `entry_hash` verifies; the failure is that it links to `ev_a68be9a6ddadeb4a` while the previous physical ledger entry is `ev_30e9f76b5a463529`. This is a real fork in the append-only chain, caused by two event-bookkeeping appends racing after they both read the same previous tip.

`ev_30e9f76b5a463529` and `ev_47347bd34f9a282d` both chained to `ev_a68be9a6ddadeb4a`. Once both lines were appended, the physical NDJSON order became linear but the hashes represented a fork.

## Root Cause

The shared helper `appendChainedJsonLine()` reads the last chained entry, computes `prev_hash`, then appends. It has no mutex or file lock. `EventManager.appendBookkeepingEvidence()` uses that helper for event bookkeeping, and `HermesLockManager.releaseTask()` / `appendEvidence()` can trigger event emissions close together.

In this incident, the `evidence.appended` bookkeeping event and `task.released` bookkeeping event both read `ev_a68be9a6ddadeb4a` as the ledger tip before either append became the new tip. Each entry is internally valid, but only one can be the next physical entry in a single linear hash chain.

## Additional Observations

A full metadata-only scan of the current local ledger found the same race class again downstream at later entries. The first verifier failure remains `ev_47347bd34f9a282d`, but additional `prev_hash` mismatches were observed after pipelined Wave 1 tool calls. That confirms this is not a one-off corrupted write; the ledger append path needs serialization.

## Impact

The ledger cannot currently prove a single total order across the affected segment. It still preserves the raw evidence entries, timestamps, owners, task IDs, and summaries, but `verifyChainedLog()` correctly returns failure because the chain forked.

No evidence payload was edited during this investigation.

## Repair vs Accept Decision

**Decision: accept and document; do not rewrite the historical ledger in place.**

Recomputing `prev_hash` and `entry_hash` for `ev_47347bd34f9a282d` and every subsequent entry would create a cleaner linear file, but it would also mutate historical attestations and destroy the original incident evidence. That is worse than a documented fork.

Recommended path:

1. Preserve the existing ledger as-is.
2. Land the P0 EvidenceLedger serialization work so future appends are single-writer.
3. Add a signed checkpoint or migration marker after the serialized ledger lands.
4. Teach verification/reporting to distinguish "historical accepted fork before checkpoint" from new post-checkpoint failures.

## Follow-Up

- P0-4/P0-5 EvidenceLedger work should serialize all chained appends behind one module.
- The verification command should eventually report all chain mismatches, not only the first break.
- Any future repair tooling must be explicit, signed, and preserve the original ledger bytes.
